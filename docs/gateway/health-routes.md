# 健康检查路由

**文件位置**: `src/routes/health.ts`

## 技术职责

该文件注册了一个单一的 HTTP GET 端点 `/health`，返回网关实例的**运行状态快照**。响应体包含以下字段：

- **status**: 固定返回 `"ok"`，表示服务存活；
- **service**: 服务标识 `"novachat-gateway"`；
- **version**: 当前网关版本号 `"0.1.0"`；
- **uptime**: 网关已运行的秒数（从进程启动时计算）；
- **onlineCount**: 当前通过此网关实例连接的 WebSocket 在线用户数；
- **timestamp**: 响应生成时的时间戳。

该端点的实现极其轻量，不涉及数据库查询或外部服务调用，确保高可用监控场景下不会因为健康检查本身导致服务过载。

## 业务角色

健康检查是**微服务架构的基础设施组件**，承担着以下关键职能：

- **负载均衡器存活探测**：Kubernetes 或 Nginx 等负载均衡器定期调用 `/health` 来判断网关实例是否存活。如果返回非 200 状态码，负载均衡器会将此实例从流量池中摘除，避免将请求路由到故障节点；
- **自动伸缩依据**：`onlineCount` 字段为水平伸缩策略提供关键信号——当在线数接近容量上限时，编排系统可以启动新的网关实例；
- **运维监控**：监控系统（如 Prometheus + Grafana）通过该端点采集网关的运行数据，生成告警和可视化面板；
- **发布流水线**：在滚动更新过程中，新版本实例的 `/health` 返回正常后，编排系统才会继续推进更新流程。

## 系统连接

- **`../ws/connection.js`** (`connectionManager`): 从 WebSocket 连接管理器中获取 `getOnlineCount()`，这是网关唯一需要从外部模块读取的数据。该字段将网关的 HTTP 层面与 WebSocket 层面连接起来，使得健康检查能反映真实的业务负载状态。
- **鉴权和限流白名单**: `/health` 被同时列入鉴权白名单（`NO_AUTH`）和限流白名单，确保监控系统随时可以访问，不受认证和限流策略干扰。
- **负载均衡器 / 容器编排系统**: 作为该端点的主要调用方，这些基础设施组件依赖健康检查结果做出流量调度决策。
