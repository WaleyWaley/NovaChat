# 令牌桶限流中间件

**文件位置**: `src/middleware/rate_limiter.ts`

## 技术职责

该文件实现了一个基于**内存令牌桶算法**的限流中间件，作为 Fastify `preHandler` 钩子注册。它为每个请求执行**双重限流校验**：

1. **IP 维度限流**：以客户端 IP 为 key，限制单个 IP 的请求速率；
2. **用户维度限流**：如果请求已通过鉴权（即 `request.userId` 已定义），额外以用户 ID 为 key 进行限流。

令牌桶的核心参数 `RATE_LIMIT_PER_USER` 和 `RATE_LIMIT_PER_IP` 从全局配置中读取，表示每秒允许的最大请求数。桶的刷新速率等于最大容量，即桶每秒恢复的数量等于其容量，确保平滑的流量整形。

内置的**清理机制**每 60 秒扫描一次桶缓存，清除超过 120 秒未活跃的条目，防止内存泄漏。

限流白名单包括 `/health` 和 PushService 内部路径，因为这些端点由负载均衡器和 C++ 后端调用，不应受客户端限流策略约束。

## 业务角色

在大规模 IM 系统中，限流是**防止滥用和保障服务质量**的第一道防线。该中间件解决了以下场景：

- **暴力破解防护**：防止攻击者通过大量登录/注册请求进行密码爆破；
- **资源公平分配**：确保单个用户或 IP 不会耗尽网关的处理能力，保障其他用户的正常使用；
- **后端保护**：C++ 微服务的处理能力有限，限流可以防止请求洪峰击穿下游服务；
- **反爬虫/反自动化**：恶意脚本通常从同一 IP 发出大量请求，IP 维度限流可直接拦截。

当限流触发时，返回 HTTP 429 状态码和业务错误码 **1501 (FLOOD_WAIT)**，客户端收到后应等待一段时间再重试。

## 系统连接

- **`../config/index.js`**: 读取 `RATE_LIMIT_PER_USER` 和 `RATE_LIMIT_PER_IP` 配置项，支持按环境调整限流阈值。
- **鉴权中间件** (`auth.ts`): 限流钩子依赖 `request.userId` 字段——该字段由鉴权中间件注入。因此在路由注册时，限流钩子必须在鉴权钩子**之后**执行。
- **健康检查路由** (`health.ts`) 和 **PushService 路由** (`push.ts`): 这两个路径被列入限流白名单，避免对内部监控和服务间通信造成干扰。
- **Phase 2 规划**：当前为内存实现，重启后计数清零；后续将升级为 Redis 令牌桶，支持多网关节点共享限流状态，实现全局统一的限流策略。
