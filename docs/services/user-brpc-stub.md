# bRPC 服务基类 — `user.brpc.h`

## 什么是 bRPC Stub

在 bRPC（基于 protobuf 的 RPC 框架）中，服务定义通常由 `.proto` 协议文件通过 `protoc-gen-brpc` 代码生成器自动产出两个类：

1. **Service 基类**（服务端基类）：定义了所有 RPC 方法的纯虚函数签名，服务端实现类继承它并覆写这些方法。bRPC 框架通过 `CallMethod` 分发请求到具体的虚函数。
2. **Stub 类**（客户端桩类）：封装了远端调用的细节，包括序列化、网络传输和反序列化。客户端通过 Stub 发起调用就像调用本地函数一样。

## 这个文件是什么

`user.brpc.h` 是 NovaChat Phase 1 阶段**手写的** `UserServiceBase` 基类，而非由 protoc 代码生成器自动生成。文件注释说明："Normally generated by protoc-gen-brpc from user.proto. Phase 1 only needs the Service base class; the _Stub is Phase 2+."

### 类结构

`UserServiceBase` 继承自 `google::protobuf::Service`，提供了：

- **12 个纯虚 RPC 方法**：`Register`、`Login`、`RefreshToken`、`Logout`、`GetUserProfile`、`GetUsers`、`UpdateProfile`、`ChangeUsername`、`CheckUsername`、`ChangePassword`、`SearchUsers`、`DeleteAccount`。每个方法的参数签名遵循 protobuf 标准模式：`(RpcController, Request, Response, Closure)`。
- **ServiceDescriptor 接口**：`GetDescriptor()` 和 `GetDescriptorStatic()` 用于服务的反射机制，bRPC 框架据此进行方法查找和路由。
- **`CallMethod` 分发**：将 `MethodDescriptor` 分发到对应的虚函数，是 bRPC 框架将网络请求路由到具体方法实现的核心桥梁。
- **原型方法**：`GetRequestPrototype` / `GetResponsePrototype` 返回每个方法的请求/响应默认实例，用于反序列化时的类型推断。

### 为什么以手写替代代码生成

Phase 1 阶段项目尚处于快速原型验证期，完整的 protoc 代码生成链路可能还未打通，手写基类可以：

- 消除对代码生成工具的依赖，加速开发迭代
- 精准控制需要暴露的方法集合
- 为 Phase 2 切换到自动生成版本预留完全兼容的接口

## 业务角色

该文件是整个 User Service RPC 通信的**契约定义层**。业务上它不包含任何实现逻辑，而是定义了 NovaChat 用户服务与外部系统之间的通信协议——客户端（TypeScript 网关或其他微服务）发送什么样的请求、期望什么样的响应，都由这个接口决定。

## 系统连接

- **上游框架**：`google::protobuf::Service` 提供基础的 RPC 反射机制，`brpc/server.h` 负责将网络请求路由到这里
- **下游实现**：`UserServiceImpl`（`user_service_impl.h`）继承 `UserServiceBase` 并提供 12 个方法的具体业务逻辑
- **协议依赖**：引用了 `nova/user/user.pb.h`（protobuf 生成的请求/响应消息类型）
- **Phase 2 扩展**：届时将切换为 protoc-gen-brpc 自动生成的文件，客户端 Stub 类将用于 TypeScript 网关层发起 bRPC 调用
